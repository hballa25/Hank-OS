# The /GOAL trick — sample prompts

> From the pack: give the model a `/goal` with a hard pass-fail line and permission to keep working until it's met. It will test its own build, catch its own lag, and iterate without you babysitting. Best for interactive parts — anything you click, drag or reload.

```
/goal make sure that moving things around and switching views in the interface never lags for me.
actually load it and check your own work - dont just assume. keep optimising until its smooth.
```

```
/goal every time i reload this it has to be ready in under ten seconds. verify by reloading it
yourself and timing it, and continuously optimise the whole thing until it consistently hits that.
```

```
/goal dont stop until the retrieval system is clearly faster and cheaper than default, verified by
real tests you run yourself - and have the test prompts ready for me so i can rerun them anytime.
```

## Hank OS adaptation

The pattern (hard pass-fail line + "verify it yourself" + "keep going until met") transfers straight to app work: "/goal TaiGrader must grade a 30-essay batch without a timeout — run it yourself and optimise until it does", or "/goal the XPScholar dashboard must load in under 3 seconds on a school Chromebook profile — throttle, measure, iterate."
